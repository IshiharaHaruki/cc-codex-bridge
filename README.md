# cc-codex-bridge (MVP)

A minimal localhost web app that bridges the **Claude Agent SDK** and the **Codex app-server** — bidirectional, user-triggered.

```
Browser ──POST /api/ask {agent, prompt}──> NDJSON event stream
  agent="claude" → runs Claude; Claude can call codex_delegate  (CC → Codex)
  agent="codex"  → runs Codex;  Codex can call ask_claude       (Codex → CC)
```

## Prerequisites

- Node 18+ (tested on 24)
- **`codex` CLI installed and logged in**: `npm i -g @openai/codex && codex login`
  - codex is both a runtime dependency (this app spawns it) and the source of the generated protocol types (see below).
- Claude logged in (Claude subscription oauth, or set `ANTHROPIC_API_KEY`)

## Run

```bash
npm install
npm run dev          # http://localhost:4399 by default
```

Open http://localhost:4399, pick an agent, type a task, and send. Both agents operate in `cwd` — the parent directory of this repo by default.

> `npm run dev` first runs the `predev` guard: if `src/codex-protocol/` is missing it auto-runs `npm run codex:protocol` to generate it. So a first `npm run dev` just works — no manual generation step.

## Codex protocol types (generated, not committed)

`src/codex-protocol/` (~600 `.ts` files) holds the Codex app-server protocol types generated by codex. They are **excluded via `.gitignore` and not in the repo** — they are generated output, strictly tied to your local codex version.

- These are **`import type` only** (pure types): erased at runtime by tsx, so the app needs none of them to run; they matter only for `tsc` type-checking.
- On first run / after clone they are generated automatically by the `predev` and `pretypecheck` guards, or manually:
  ```bash
  npm run codex:protocol      # = codex app-server generate-ts --out src/codex-protocol --experimental
  ```
- **After upgrading codex**, regenerate once. If the types no longer match the new protocol, `npm run typecheck` will surface it immediately.

## Type-check

```bash
npm run typecheck    # auto-generates types if missing, then runs tsc --noEmit
```

## How it works

| File | Role |
|---|---|
| `src/codex-client.ts` | Spawns `codex app-server`, speaks JSON-RPC over stdio, drives thread/turn, streams `agentMessage`. Dispatches Codex's `item/tool/call` requests to local handlers via `dynamicTools` (this is how `ask_claude` works). Adapted from boop-agent (Apache-2.0). |
| `src/claude-runtime.ts` | Wraps `query()`; when `withCodexTool` is set, injects the in-process `codex_delegate` tool (CC→Codex). |
| `src/server.ts` | HTTP server + NDJSON stream; injects the `ask_claude` dynamic tool on the Codex side (Codex→CC). |
| `src/codex-protocol/` | Protocol types from `codex app-server generate-ts` (**generated, not committed** — see above; do not hand-edit). |
| `public/index.html` | Minimal single-page UI. |

## Configuration

- Port: `PORT=4399`
- Working directory: `BRIDGE_CWD=/abs/path`
- **Sandbox**: the MVP defaults Codex to `read-only` (it can review/analyze/answer, but not edit files). To let Codex actually modify code, change `sandbox` to `"workspace-write"` in `src/server.ts` and `src/claude-runtime.ts`.
- **Auth (env-driven — set a key to use the API, otherwise subscription)**:
  - `ANTHROPIC_API_KEY=sk-ant-...` → Claude uses API billing (otherwise Claude subscription oauth)
  - `OPENAI_API_KEY=sk-...` → Codex uses API billing (otherwise ChatGPT subscription; the client auto-adds `-c preferred_auth_method="apikey"` to force it)

## Notes & limitations

- Requires `@anthropic-ai/claude-agent-sdk@^0.3` (0.1.x has a `tool_use ids must be unique` bug with in-process MCP tools across turns; needs zod v4).
- Cross-calls are request/response. For two agents holding a multi-turn autonomous conversation you'd add busy-guard / steer / interrupt (see quilin-ai/agent-bridge).
- Each Codex call spawns a fresh app-server (one-shot). For lower latency, switch to a persistent connection.
